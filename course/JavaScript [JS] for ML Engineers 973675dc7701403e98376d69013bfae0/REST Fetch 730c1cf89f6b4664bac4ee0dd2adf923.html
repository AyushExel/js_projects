REST/ Fetch

In very brief, let us quickly understand the role of server or backend in our applications.

You can think of the server as a wrapper around the actual database of the application. The database contains all the useful information needed for the application. In a broader sense, this structured information and its state is referred to as resources.

Resources and Representations

A resource can be a web page, a video stream, or an image, for example. A resource can even be an abstract concept, such as the list of all users in a database or the weather forecast for a particular location. The only real constraint is that every resource in a system is uniquely identifiable. the server is responsible for managing the state of the resource, but a client can choose which representation they prefer to interact with.

What Is REST?

REST stands for Representational State Transfer. Restful Web Services is a lightweight, maintainable, and scalable service that is built on the REST architecture. Restful Web Service, expose API from your application in a secure, uniform, stateless manner to the calling client. The calling client can perform predefined operations using the Restful service. The underlying protocol for REST is HTTP.

Attributes that define RESTful systems

Uniform Interface

The uniform interface is one of the distinctive attributes of RESTful systems. It requires clients to access all resources using the same set of standard operations.

Stateless

All operations in a RESTful system should be stateless. This means each call from the client to the server must not rely on any shared state. It also implies that every request to the server must include all required data for the server to fulfill the request.

Restful Methods

Let’s assume that we have a RESTful web service is defined at the location. http://myapp.com/employee . When the client makes any request to this web service, it can specify any of the normal HTTP verbs of GET, POST, DELETE and PUT. Below is what would happen If the respective verbs were sent by the client.

  1. POST – This would be used to create a new employee using the RESTful web service
  1. GET – This would be used to get a list of all employees using the RESTful web service
  1. PUT – This would be used to update all employees using the RESTful web service
  1. DELETE – This would be used to delete all employees using the RESTful services

Let’s take a look from a perspective of just a single record. Let’s say there was an employee record with the employee number of 1.

The following actions would have their respective meanings.

  1. POST – This would not be applicable since we are fetching data of employee 1 which is already created.
  1. GET – This would be used to get the details of the employee with Employee no as 1 using the RESTful web service
  1. PUT – This would be used to update the details of the employee with Employee no as 1 using the RESTful web service
  1. DELETE – This is used to delete the details of the employee with Employee no as 1

These actions are assigned to a specific link like  http://myapp.com/employee/ are also called endpoints of a server

Fetch

All of the previous section was merely an intro to REST APIs. The section wasn’t intended to help you design the APIs but to consume them.

the fetch function is the most common way to consume resources asynchronously. It uses promises.

Reading Material:

Let’s take a look at the executable example from the article.

Example - Using GitHub public API

  • Execute the following code in your browser. Put this script in a separate file or in the html boiler plate itself.
// GET Request.
fetch("https://api.github.com/users/manishmshiva")
  // Handle success
  .then((response) => response.json()) // convert to json
  .then((json) => console.log(json)) //print data to console
  .catch((err) => console.log("Request Failed", err)); // Catch errors
  • Open the console from dev options and you should see a response json object.
  • Now, replace the last part of the link with your username. In the response, you’ll be able to see the link to your public avatar

There are many useful public APIs available. Some require many levels of authentication. For example, check out twitter public api
